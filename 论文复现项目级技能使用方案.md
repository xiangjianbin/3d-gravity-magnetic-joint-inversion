## 一、整体架构：Skill + Task Agent 混合调度

### 1.1 核心设计哲学

```
┌─────────────────────────────────────────────────────────────┐
│                    主会话（永远轻量）                          │
│                                                             │
│  ✅ 读每个阶段的 .md 报告（摘要级别）                         │
│  ✅ 判断 pass/fail/need-fix                                 │
│  ✅ 决定下一个阶段启动哪些 Agent                              │
│  ✅ 在 Agent 之间通过文件路径传递信息                         │
│  ✅ 处理异常决策                                             │
│  ✅ 维护 REPRODUCTION_STATE.json 全局状态                    │
│  ✅ 每个 Phase 结束后 git commit                             │
│                                                             │
│  ❌ 不读完整源代码                                           │
│  ❌ 不读完整训练日志                                         │
│  ❌ 不替 Agent 写代码                                        │
│  ❌ 同时不超过 3 个活跃 Agent                                 │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 Skill vs Agent 的分工

| 维度 | Skill（现有技能） | Task Agent（子代理） |
|------|-------------------|---------------------|
| **触发方式** | `/skill-name` 或 `Skill("skill-name")` | `Agent({prompt: "...", subagent_type: "general-purpose"})` |
| **适用场景** | 有成熟流程的标准化任务 | 需要写代码/调试的定制任务 |
| **上下文** | 加载 SKILL.md 后在主会话执行 | 完全独立的上下文，做完销毁 |
| **输入** | `$ARGUMENTS` 参数 | 自定义 prompt + 文件路径 |
| **输出** | 按 SKILL.md 规定的文件 | 返回精简摘要 + 写 .md 报告 |
| **在本 pipeline 中的角色** | 分析类、绘图类、审计类任务 | 代码编写、数据生成、模型实现、训练执行 |

### 1.3 调度方式详解

#### 方式 A：直接调用 Skill（主会话内执行）

适用于：文本分析、结果解读、图表生成、实验审计等不涉及大量代码的任务。

```python
# 示例：分析实验结果
Skill("analyze-results", args="results/metrics.json")

# 示例：生成图表
Skill("paper-figure", args="figures/plan.md")

# 示例：实验审计
Skill("experiment-audit", args="results/")
```

**触发时机**: 主会话判断当前阶段是"分析/绘图/审计"类型时，直接调用。

#### 方式 B：Task Agent（独立上下文执行）

适用于：写代码、调试、运行训练等需要大量代码上下文的任务。


**触发时机**: 主会话判断当前阶段需要"写代码/跑代码"时，启动 Agent。

## 二、Pipeline 总览 — 8 个 Phase

```
Phase 0: 项目初始化 + GitHub 连接
    │  主会话直接做（配置类，不涉及大代码）
    ▼
Phase 1: 论文深度解析 → PAPER_ANALYSIS.md
    │  主会话 + Skill("research-lit") 辅助
    ▼
Phase 2: 数据集生成 + 验证 + 可视化
    │  Task Agent × 2（并行：正演模块 + 数据管道）
    │  → DATASET_REPORT.md + figures/dataset_*.png
    ▼
Phase 3: 模型架构实现 + 冒烟测试
    │  Task Agent × 3（并行：骨干网络 + 任务头 + 集成测试）
    │  → MODEL_REPORT.md + SMOKE_TEST_REPORT.md
    ▼
Phase 4: GPU 检查 + 训练配置优化
    │  轻量 Agent（nvidia-smi + 显存估算）
    │  → GPU_ALLOC_PLAN.md
    ▼
Phase 5: 训练执行 + 自动修复
    │  后台 Bash Agent 跑训练 + 按需修复 Agent
    │  → TRAINING_LOG.md + checkpoints/
    ▼
Phase 6: 结果对齐 + 图表生成
    │  Task Agent（数值对比）+ Skill("paper-figure")
    │  → RESULT_COMPARISON.md + figures/result_*.pdf
    ▼
Phase 7: 实验审计 + 最终报告
    │  Skill("experiment-audit") + Skill("result-to-claim")
    │  → EXPERIMENT_AUDIT.md + FINAL_REPORT.md
```

**ASSUMPTIONS_AND_GAPS.md 结构**:
```markdown
# 假设与 Gap 清单

## Gap 1: 解码器通道数
- 论文只给了编码器通道数，解码器按对称推断
- **假设**: 解码器通道 1024→512→256→128→64

## Gap 2: LeakyReLU 的 alpha/gamma
- 论文未给出具体数值
- **假设**: alpha=0.01, gamma=0.01（标准默认值）

## Gap 3: 学习率调度参数
- Eq.12 给出了公式形式但未给出 β₀,β₁,β₂,γ,lr₀ 具体值
- **假设**: lr₀=1e-3, beta_0=0.5, beta_1=0.999, beta_2=0.99（Adam 默认 + 余弦退火变体）

## Gap 4: Batch Size
- 论文未明确给出
- **假设**: 从显存倒推，32GB 可用 → batch_size=4~8

## Gap 5: ASPP 各分支 filter 数
- 论文未明确
- **假设**: 每分支 256 filters

## Gap 6: 径向网格操作 (Radial Grid Operation)
- 这是论文的创新组件，描述较模糊
- **策略**: 先实现标准版本，再根据效果微调

## Gap 7: 地磁参数（合成数据）
- 论文未指定合成数据的 I, D, F 值
- **假设**: I=45°, D=0°, F=50000 nT（典型中纬度值）
```

### Phase 3: 模型架构实现 + 冒烟测试

**拆分为 3 个 Agent（可部分并行）**

#### Agent 3A: 骨干网络 (Backbone U-Net)

**Prompt 要点**:
```
你是 3D 深度学习架构专家。

【前置阅读】
1. docs/PAPER_ANALYSIS.md — Section 2 网络架构细节
2. docs/ASSUMPTIONS_AND_GAPS.md — 架构相关假设

【任务】
实现 src/model/backbone_unet3d.py:

架构规格（来自论文 Fig.2）:
- 输入: [B, 2, 40, 40, 20] （重力+磁异常 2通道）
- 编码器 4 层:
  Level 1: Conv3d(2→64, k=3, s=1, p=1) → LeakyReLU → MaxPool3d(2) → [B,64,20,20,10]
  Level 2: Conv3d(64→128, k=3) → LeakyReLU → MaxPool3d(2) → [B,128,10,10,5]
  Level 3: Conv3d(256→256, k=3) → LeakyReLU → MaxPool3d(2) → [B,256,5,5,2]
  Level 4: Conv3d(256→512, k=3) → LeakyReLU → MaxPool3d(2) → [B,512,2,2,1]
  Bottleneck: Conv3d(512→1024, k=3) → LeakyReLU → [B,1024,2,2,1]
- 解码器 4 层（对称 + skip connection）:
  使用 Upsample + Concat + Conv
- 输出: [B, C, 40, 40, 20]（C 由 ASPP 决定）

额外实现:
- src/model/aspp.py — ASPP 模块（Fig.4）
  dilation rates = [6, 12, 18]
  每分支: Conv3d(256, 256, k=3, dilation=d) → BN → ReLU
  全局平均池化分支: AdaptiveAvgPool3d(1) → Conv3d(256,256,k=1) → Upsample
  Concat 所有分支 → Conv3d(256*5, 256, k=1)

【约束】
- 必须支持 torch.cuda.amp 混合精度
- 使用 gradient checkpointing 节省显存（虽然 32GB 够用，但保持习惯）
- 所有卷积使用 padding='same' 保持尺寸

【输出】
- src/model/backbone_unet3d.py
- src/model/aspp.py
- tests/test_backbone.py（前向传播测试：输入随机张量，检查输出形状）
- docs/BACKBONE_REPORT.md
- 返回精简摘要
```

#### Agent 3B: 多任务头 (Multi-task Heads)

**Prompt 要点**:
```
你是多任务学习架构专家。

【前置阅读】
1. docs/PAPER_ANALYSIS.md — Tasks 1-5 详细描述
2. docs/BACKBONE_REPORT.md — Backbone 已就绪

【任务】
实现以下 5 个任务头:

1. src/model/task_independent_gravity.py — Task 1: 独立重力反演
   - 输入: backbone 特征 [B, 256, 40, 40, 20]
   - 4 层解码器结构（类似小 U-Net）
   - 输出: M_rho_pred [B, 1, 40, 40, 20]
   - Loss: MSE

2. src/model/task_independent_magnetic.py — Task 2: 独立磁法反演
   - 同 Task 1 结构
   - 输出: M_kappa_pred [B, 1, 40, 40, 20]
   - Loss: MSE

3. src/model/task_structural_similarity.py — Task 3: 结构相似性提取
   - 输入: [M_rho_pred, M_kappa_pred] 拼接
   - Conv3d → MaxPool3d(2) ×2 → 径向网格操作 → Conv3d → MaxPool3d
   - 输出: S [B, 1, 40, 40, 20]，Sigmoid 激活
   - Loss: MSE

4. src/model/task_joint_gravity.py — Task 4: 联合重力反演
   - 输入: 原始输入 [B,2,40,40,20] + S [B,1,40,40,20] → concat → [B,3,40,40,20]
   - 类似 Task 1/2 的解码器结构
   - 输出: M_rho_final [B, 1, 40, 40, 20]
   - Loss: BCE (omega=1)

5. src/model/task_joint_magnetic.py — Task 5: 联合磁法反演
   - 同 Task 4 结构
   - 输出: M_kappa_final [B, 1, 40, 40, 20]
   - Loss: BCE (omega=1)

6. src/model/joint_inversion_net.py — 完整网络组装
   - 组装所有模块: Backbone → ASPP → Task1-5
   - forward() 返回所有 5 个任务的输出
   - 支持推理模式（只返回 Task 4+5）

7. src/model/loss_functions.py — 损失函数组合
   - Total_Loss = MSE_1 + MSE_2 + MSE_3 + BCE_4 + BCE_5
   - 支持各任务权重调整

【输出】
- 以上所有 .py 文件
- tests/test_heads.py（形状测试 + loss 计算测试）
- docs/MODEL_REPORT.md（架构总结 + 参数量统计）
- 返回精简摘要
```

```
5. 跑冒烟测试: python src/train.py --config configs/smoke.yaml
6. 验证:
   - 前向传播无报错
   - 反向传播无报错
   - 一步 optimizer.step() 成功
   - 显存占用 < 28GB（留余量）
   - 输出指标格式正确

【输出】
- src/train.py, src/evaluate.py, src/utils.py
- configs/smoke.yaml, configs/full.yaml
- docs/SMOKE_TEST_REPORT.md
- 返回精简摘要（pass/fail + 关键数据）
```

**Phase 3 判定标准**:
- [ ] 所有模块前向传播形状正确
- [ ] 冒烟训练 2 epochs 无报错
- [ ] 显存 < 28GB
- [ ] 参数量在合理范围（预计 50M-200M）
- [ ] 损失函数能下降（证明梯度流通）

### Phase 5: 训练执行 + 自动修复

**这是最关键的 Phase，采用"后台运行 + 按需修复"模式**

#### 5.1 启动后台训练

```bash
# 主会话启动后台训练
CUDA_VISIBLE_DEVICES=0 python src/train.py --config configs/full.yaml \
    2>&1 | tee logs/training.log &
```

使用 `Bash(run_in_background=True)` 执行。

#### 5.2 训练监控（定期检查）

**方式 A**: 主会话每 10 分钟用 Cron/ScheduleWakeup 检查一次日志尾部

**方式 B**: 派轻量 Agent 检查

```
监控 Agent prompt:
读取 logs/training.log 最后 100 行。
检查:
1. 是否有 ERROR / Exception / Traceback
2. loss 是否为 NaN 或 Inf
3. loss 是否在下降（对比最近 10 个 epoch）
4. GPU 利用率（如果有 nvidia-smi 日志）
5. 当前进度（epoch/total_epochs）

返回: status=OK/WARNING/CRITICAL + 一句话诊断
```

#### 5.3 自动修复机制

当监控发现 CRITICAL 问题时的流程：

```
训练崩溃/NaN/OOM
    │
    ▼
主会话收到 CRITICAL 报告
    │
    ├──→ 派"修复 Agent"
    │     prompt: "读取 logs/training.log 错误信息 + 相关源码
    │           定位根因 → 修复 → 跑最小测试验证
    │           最大重试 3 次"
    │
    ├──→ 修复成功 → 重新启动后台训练（从 checkpoint 或从头）
    │
    └──→ 修复失败 3 次 → 暂停，报告人工决策
```

**修复 Agent Prompt 模板**:
```
你是深度学习调试专家。训练出现了错误。

【错误信息】
{从 logs/training.log 提取的最后 50 行}

【相关代码】
- src/model/joint_inversion_net.py
- src/train.py
- configs/full.yaml

【任务】
1. 定位错误的根本原因
2. 修复代码
3. 用 configs/smoke.yaml 跑 2 步验证修复有效
4. 如果是 OOM: 降低 batch_size 或启用更多 gradient checkpointing
5. 如果是 NaN: 检查 learning rate / loss scale / 数据范围
6. 如果是 shape mismatch: 打印所有中间张量形状

【约束】
- 最大重试次数: 3
- 每次修复必须跑通 smoke test
- 修复记录写入 docs/FIX_LOG.md

【输出】
- 修复了什么（一句话）
- smoke test 是否通过
- 是否需要重新开始训练
```

## 六、防止上下文爆炸的策略

### 6.1 主会话上下文预算

| 操作 | 预估 token 消耗 | 频率 |
|------|-----------------|------|
| 读一个 .md 报告（摘要级） | ~2K tokens | 每个 Phase 1-2 次 |
| 判断 pass/fail | ~500 tokens | 每个 Phase 1 次 |
| 启动 Agent（发送 prompt） | ~3K tokens | 每个 Phase 1-3 次 |
| 接收 Agent 摘要 | ~1K tokens | 每个 Agent 完成时 |
| git commit | ~200 tokens | 每个 Phase 1 次 |
| **每个 Phase 净增** | **~8-15K tokens** | |

按 8 个 Phase 计算，总消耗约 **64K-120K tokens**，远低于上下文窗口上限（200K tokens）。

### 6.2 Agent 上下文隔离

每个 Task Agent 是完全独立的会话：
- 不继承主会话历史
- 只通过文件获取上游信息
- 完成后销毁，释放资源
- 即使单个 Agent 上下文爆炸也不影响主会话

### 6.3 文件大小控制

| 文件 | 大小限制 | 超过时处理 |
|------|---------|-----------|
| PAPER_ANALYSIS.md | ≤ 300 行 | 拆分为详细版 + 摘要版 |
| DATASET_REPORT.md | ≤ 200 行 | 只保留统计摘要 |
| TRAINING_LOG.md | ≤ 150 行 | 只保留关键节点表格 |
| RESULT_COMPARISON.md | ≤ 250 行 | 表格为主，省略原始数据 |
| FINAL_REPORT.md | ≤ 200 行 | 摘要式 |
| *.py 源代码 | 不限 | 主会话不读 |

### 6.4 并发控制

| 场景 | 最大并发 Agent 数 | 原因 |
|------|-------------------|------|
| Phase 2 (数据集) | 2 | 正演模块 + 数据管道可并行 |
| Phase 3 (模型) | 3 | Backbone + Heads + 集成（集成依赖前两个） |
| Phase 5 (训练) | 1 (后台) + 1 (修复) | 训练占 GPU，修复等 GPU 空闲 |
| 其他 Phase | 1 | 串行即可 |

## 八、风险与缓解措施

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 正演公式实现有误 | 中 | 高 | 用已知解析解验证；对比开源库 |
| 32GB 仍不够 batch_size>1 | 低 | 中 | 自动降 batch_size；gradient checkpointing |
| 训练不收敛 | 中 | 高 | 监控 Agent 早发现；调 LR/检查数据 |
| Agent 上下文爆炸 | 低 | 中 | prompt 精简控制；分阶段执行 |
| 论文 Gap 导致结果偏差大 | 高 | 中 | 记录所有假设；容差 ±10% |
| GitHub push 失败 | 低 | 低 | 重试 3 次；检查 token 权限 |
| 数据集生成太慢（45000组） | 高 | 中 | 先生成小规模验证；大规模后台生成 |
