# 论文复现 Pipeline：3D Joint Inversion of Gravity and Magnetic Data

> **论文**: Improved_3D_Joint_Inversion_of_Gravity_and_Magnetic_Data_Based_on_Deep_Learning_With_a_Multitask_Learning_Strategy
> **GPU**: RTX 3060 Laptop (6GB VRAM, ~4GB 可用)
> **策略**: Skill + Task Agent 混合调度，主会话轻量监督

---

## 一、总体架构设计

### 1.1 设计原则（来自 Task Agent 思路.md）

| 原则 | 实现方式 |
|------|----------|
| 主会话永远轻量 | 只做调度 + 读文档 + 判断下一步，不写代码 |
| 能用 Skill 就用 Skill | 数据分析→/analyze-results，画图→/paper-figure，审计→/experiment-audit |
| 写代码用 Task Agent | 上下文隔离，做完销毁，返回精简摘要 |
| Agent 数量动态决定 | 简单任务 1 个 Agent，复杂任务拆成多个并行 |
| 跨 Agent 信息通过文件传递 | .md/.json 文件作为唯一通信渠道 |
| Agent 返回精简摘要 | 不返回全部代码/日志 |

### 1.2 Pipeline 总览（6 个 Phase）

```
Phase 0: 项目初始化 + GitHub 连接
    ↓  输出: CLAUDE.md, README.md, Git 仓库建立
Phase 1: 论文解析 + 实验计划生成
    ↓  输出: PAPER_ANALYSIS.md, EXPERIMENT_PLAN.md
Phase 2: 数据集自动生成 + 验证 + 可视化
    ↓  输出: data/, DATASET_REPORT.md, data_viz/
Phase 3: 模型实现 + 训练 + 自动修复
    ↓  输出: src/model.py, src/train.py, checkpoints/, TRAINING_LOG.md
Phase 4: 结果生成 + 与论文对比 + 图表绘制
    ↓  输出: results/, FIGURE_COMPARISON.md, figures/
Phase 5: 实验审计 + 最终报告 + Git 提交
    ↓  输出: EXPERIMENT_AUDIT.md, FINAL_REPORT.md
```

### 1.3 文档传递链（核心机制）

```
Agent-A ──写──→ doc_A.md ──读──→ Agent-B ──写──→ doc_B.md ──读──→ 主会话判断
                 ↑                    ↑                    ↑
            唯一通信渠道        唯一通信渠道          读所有doc做决策
```

每个 Agent 完成后必须输出：
1. **状态报告** (`*_REPORT.md`): 做了什么、结果如何、遇到什么问题
2. **数据产物** (代码/数据/图表): 实际产出物
3. **下一步建议** (`_REPORT.md` 末尾): 给主会话的决策建议

### Phase 2: 数据集自动生成 + 验证 + 可视化

**目标**: 按照论文规格生成合成数据集，验证正确性，可视化各类数据

**执行方式**: 多个 Task Agent 并行 + Skill 组合

#### 2.1 数据生成 Agent

- **名称**: `data-generator`
- **类型**: general-purpose（需写代码）
- **输入**: `docs/PAPER_ANALYSIS.md` 中的数据集规格
- **输出**: `data/` 目录 + `docs/DATASET_REPORT.md`

**数据生成代码要点** (基于论文):

```python
# 核心数据生成逻辑（Agent 需要实现的）
1. 地质体建模：
   - 随机生成 3D 地下密度模型 ρ(x,y,z) 和磁化率模型 κ(x,y,z)
   - 几何形状：棱柱体、球体、不规则体
   - 参数范围：按论文 Section 3 的描述

2. 正演计算：
   - 重力异常 Δg = G * ∫∫∫ ρ(r')/|r-r'|³ dV'
   - 磁异常 ΔT = μ₀/4π * ∫∫∫ M·∇(1/|r-r'|) dV'
   - 使用论文中相同的正演公式

3. 数据格式：
   - 输入：[Δg_grid, ΔT_grid] → 2通道 3D 体数据
   - 标签：[ρ_model, κ_model] → 2通道 3D 体数据
   - 维度：按论文规定的网格大小

4. 数据划分：
   - Training / Validation / Test = 论文比例
   - 固定随机种子保证可复现

5. 噪声添加：
   - 高斯噪声，σ 按论文设置
   - 多个噪声水平用于鲁棒性实验
```

**自动修复机制**:
```
运行数据生成脚本
  ├─ 成功 → 进入验证阶段
  └─ 失败 → 自动诊断（OOM/ImportError/数值错误）
       ├─ OOM → 减小网格分辨率或分批生成
       ├─ ImportError → pip install 缺失包
       └─ 数值错误 → 检查正演公式实现
       → 最多重试 3 次
```

### Phase 3: 模型实现 + 训练 + 自动修复

**目标**: 实现 3D U-Net 多任务模型，完成训练，自动处理错误

**GPU 约束**: RTX 3060 6GB → 需要：
- 控制 batch size（可能需要 1-2）
- 使用 gradient checkpointing
- 可能需要减小模型或网格分辨率
- 使用 mixed precision (AMP)

#### 3.1 模型架构 Agent

- **名称**: `model-architect`
- **类型**: general-purpose
- **输入**: `docs/PAPER_ANALYSIS.md` 的模型章节 + GPU 信息
- **输出**: `src/model/` 所有文件 + `docs/MODEL_REPORT.md`

**必须实现的模块**:

```
src/model/
├── unet3d.py              # 3D U-Net 主体
│   ├── Encoder: 4层下采样 (3×3×3 conv + BN + ReLU + MaxPool)
│   ├── Bottleneck: 3×3×3 conv
│   └── Decoder: 4层上采样 (TransposeConv + Skip + Conv)
├── multitask_head.py      # 双任务输出头
│   ├── Gravity head: 1×1×1 conv → density prediction
│   └── Magnetic head: 1×1×1 conv → susceptibility prediction
└── loss_functions.py      # 损失函数
    ├── MSE Loss (per task)
    ├── Multi-task weighted loss
    └── Uncertainty weighting (Kendall et al., 2018) 如论文使用
```

**GPU 适配策略** (Agent 必须考虑):
```python
# 显存优化措施
1. batch_size = 1 或 2（根据实际测试调整）
2. torch.cuda.amp.autocast()  # 混合精度
3. torch.utils.checkpoint.checkpoint  # 梯度检查点
4. 如仍 OOM → 减少基础通道数 (64→32→16)
5. 如仍 OOM → 减少 encoder 层数 (4→3)
```

#### 3.2 训练代码 Agent

- **名称**: `trainer-coder`
- **类型**: general-purpose
- **与 model-architect 关系**: 读取 MODEL_REPORT.md 了解接口
- **输出**: `src/train.py`, `src/evaluate.py`, `src/utils.py`, `configs/*.yaml`

**训练脚本必须包含**:

```python
# train.py 核心功能
1. argparse 接口（所有超参可配）
2. DataLoader 设置（num_workers, pin_memory）
3. 训练循环：
   - epoch loop → batch loop
   - 前向传播 → loss 计算 → 反向传播 → 优化器步进
   - 每 N 步记录 log
   - 验证集评估
   - 最佳模型保存
4. 指标计算：MSE, RMSE, 相关系数 (Correlation Coefficient)
5. Checkpoint 保存/加载
6. 固定随机种子
7. WandB 日志（可选）
8. 结果保存为 JSON（供后续 analyze-results 使用）
```

#### 3.3 Sanity Check + 自动修复循环

**这是 Phase 3 的核心质量控制环节**:

```
                    ┌──────────────────────┐
                    │  启动 sanity training  │
                    │  (小数据 + 少 epoch)   │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │    检查训练输出        │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
         ✅ PASS           ❌ FAIL: OOM      ❌ FAIL: 其他
              │                │                │
              ▼                ▼                ▼
      进入完整训练      ┌─────────────┐   ┌─────────────┐
                      │ fix Agent    │   │ fix Agent    │
                      │ (显存优化)    │   │ (错误修复)   │
                      └──────┬───────┘   └──────┬───────┘
                             │                  │
                             └────────┬─────────┘
                                      ▼
                              重新 sanity (最多3次)
                                      │
                               3次都失败?
                                │     │
                               NO    YES → 报告给主会话
```

**Sanity Check 标准**:
- [ ] 训练无报错跑完
- [ ] Loss 在下降（不是 NaN 或发散）
- [ ] GPU 利用率 > 50%（不是空闲）
- [ ] 输出了 checkpoint 和 metrics JSON
- [ ] 评估指标在合理范围内

#### 3.4 完整训练执行

- **方式**: 后台 Bash Agent (`run_in_background: true`)
- **监控**: 定期轻量 Agent 检查训练状态
- **输出**: `checkpoints/best_model.pt`, `docs/TRAINING_LOG.md`

**训练监控 Agent** (轻量):
- **名称**: `training-monitor`
- **类型**: Bash（仅运行 nvidia-smi 和检查日志）
- **频率**: 每 5 分钟一次（由主会话调度）
- **检查内容**: GPU 占用、loss 最新值、预计剩余时间

**自动修复 Agent** (按需):
- **名称**: `code-fixer`
- **类型**: general-purpose
- **触发**: monitor 发现异常时由主会话启动
- **能力**: 读错误日志 → 诊断 → 改代码 → 重启训练

#### 3.5 Phase 3 协调时序

```
主会话调度:
  ① 启动 model-architect Agent
       ↓ → src/model/* + MODEL_REPORT.md
  ② 主会话读 MODEL_REPORT → 检查架构合理性
  ③ 启动 trainer-coder Agent（读 MODEL_REPORT）
       ↓ → src/train.py + configs/ + utils
  ④ 主会话快速 review 代码（不深入细节）
  ⑤ 启动 sanity check (后台 Bash)
       ↓ → 成功/失败
  ⑥ 如果失败 → 启动 code-fix Agent → 回到 ⑤
  ⑦ sanity 通过 → 启动完整训练 (后台 Bash)
  ⑧ 周期性启动 training-monitor (轻量)
  ⑨ 训练完成 → 读 TRAINING_LOG.md
  ⑩ Git commit: "phase 3: model trained successfully"
```

## 四、上下文管理策略（防溢出）

### 4.1 主会话上下文预算

| 操作 | 预估 token | 频率 |
|------|-----------|------|
| 读 Agent 报告 (~2KB) | ~500 | 每个 Phase 结束 |
| 做 Go/No-Go 决策 | ~200 | 每个 Phase 结束 |
| 启动新 Agent | ~300 | 每个 Phase 开始 |
| Git 操作 | ~100 | 每个 Phase 结束 |
| **每 Phase 净增** | **~1100** | - |
| **总 6 Phase** | **~6600** | 远低于限制 |

### 4.2 Agent 上下文隔离

- 每个 Agent 是独立进程
- Agent 内部上下文不影响主会话
- Agent 完成后销毁，释放资源
- 只有精简报告传回主会话

### 4.3 大文件处理

- 代码文件: Agent 内部处理，不回传
- 数据文件: 存磁盘，不加载到上下文
- 图表文件: 存磁盘，主会话按需 Read 查看
- 日志文件: 只读最后 N 行

### 4.4 长训练处理

- 训练在后台 Bash 运行
- 主会话不等待训练完成
- 通过定期读日志文件监控
- 日志文件滚动，只保留最新

## 五、错误恢复策略

### 5.1 Agent 级错误恢复

```
Agent 执行失败
  ├─ 第 1 次: 自动重试（相同参数）
  ├─ 第 2 次: 调整参数重试（如减小 batch_size）
  ├─ 第 3 次: 启动专门的 fix-agent 诊断
  └─ 3 次都失败: 写入错误报告，通知主会话
```

### 5.2 Phase 级错误恢复

```
Phase 失败
  ├─ 数据问题: 回到 Phase 2 重新生成
  ├─ 模型问题: 回到 Phase 3 重新实现
  ├─ 训练崩溃: 从最近 checkpoint 恢复
  └─ 结果不符: 分析原因，局部重跑
```

### 5.3 Git 级恢复

```
每个 Phase 结束都 commit
  → 任何时刻都可以回到上一个稳定状态
  → 不会丢失已完成的工作
```

## 七、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| GPU 显存不足 | 高 | 高 | 动态 batch size + AMP + gradient checkpointing |
| 正演公式实现误差 | 中 | 高 | 仔细对照论文公式 + 物理一致性验证 |
| 论文超参数不全 | 中 | 中 | 用合理默认值 + 在报告中注明 |
| 训练不收敛 | 低 | 高 | 学习率调度 + 早停 + 多种初始化 |
| 上下文溢出 | 低 | 中 | 严格文档传递 + Agent 隔离 |
| 随机种子差异 | 中 | 低 | 固定种子 + 报告偏差范围 |
